## Design Context

### Users
The primary users are technical developers and backend-oriented platform operators. They use the product while configuring modules, integrations, spaces, security, and data features across a Unified Data Platform. Their main job is to enable or adjust platform capabilities quickly without fighting the UI.

### Brand Personality
Unified, capable, backend-first.

The interface should feel confident, structured, and efficient. It should communicate that the platform is powerful and technical, but still easy to configure and approachable for day-to-day operational work.

### Aesthetic Direction
The visual direction should take inspiration from Supabase in the parts that feel clean, modern, and developer-friendly, but this product should remain more platform-oriented and configuration-first rather than product-marketing-first.

The UI should:
- feel like a Unified Data Platform and AI platform
- emphasize clear information hierarchy and low-friction configuration
- avoid looking like a heavy enterprise cloud console with intimidating density
- support both light and dark themes as first-class experiences
- use blue and black as the dominant brand anchors, with restrained accents

Anti-direction:
- overly complex cloud-service control-panel aesthetics
- noisy AI-generated visual tropes
- decorative UI that slows configuration work

### Design Principles
1. Configuration first.
Every screen should help technical users understand and change system behavior quickly, with minimal friction and clear defaults.

2. Backend power, simplified.
Expose advanced capability without making the interface feel intimidating or overloaded.

3. Structured over flashy.
Use strong hierarchy, spacing, and contrast to guide decisions. Visual polish should support comprehension, not distract from it.

4. Developer-trustworthy.
Interfaces should feel precise, consistent, and reliable, like a tool built for people who care about systems and correctness.

5. Unified platform language.
AI, data, security, spaces, and configuration modules should feel like parts of one coherent system rather than separate products.
